B.1  CUSTOMER-FACING RESTAURANT LIST + SEARCH
---------------------------------------------
Currently the backend has no `GET /api/restaurants` listing endpoint and no
search. The Flutter discovery feed needs:

  GET /api/restaurants
        Query params:
          q             (string, optional — text query)
          lat, lng      (float, optional — for distance scoring)
          cuisine       (string, optional)
          limit         (int, default 25, max 50)
          offset        (int, default 0)
        Response:
          { "restaurants": [ Restaurant (with computed `distance_km` if lat/lng) ],

  GET /api/restaurants/search
        Same params as above plus full-text + ranking. Wraps OpenSearch when
        configured, falls back to Mongo `$text` index when not.

  GET /api/restaurants/search/suggest?q=<prefix>
        Returns `{suggestions: [string]}`. Pulls top N restaurant names &
        cuisines matching the prefix.

  POST /api/ai/search
        body: { "query": "spicy thai under 30 min near me",
                "lat": ..., "lng": ... }
        resp: { "intent": { cuisine?, max_delivery_minutes?,
                            dietary_tags?, price_band?, ... },
                "restaurants": [ Restaurant ] }
        Server-side: call an LLM (or rule-based parser if no LLM key) to extract
        intent → translate to OpenSearch query → return ranked restaurants.

Add these files:
  internal/apps/discovery/controller/controller.go
  internal/apps/discovery/service/service.go
  internal/apps/discovery/repository/repository.go
  internal/apps/discovery/opensearch/client.go        (new dep)

Register routes in internal/server/routes.go inside the public `api` group:
  api.Get("/restaurants", discoveryCtl.List)
  api.Get("/restaurants/search", discoveryCtl.Search)
  api.Get("/restaurants/search/suggest", discoveryCtl.Suggest)
  api.Post("/ai/search", discoveryCtl.AISearch)

Add Mongo indexes in internal/database/database.go:
  restaurants: { name: "text", description: "text", cuisine_tags: "text" }
  restaurants: { location: "2dsphere" }   // requires storing GeoJSON Point

Add fields to Restaurant model (internal/apps/restaurant/model/model.go):
  CuisineTags          []string  `bson:"cuisine_tags"`
  Location             GeoJSON   `bson:"location,omitempty"`   // Point
  AverageRating        float64   `bson:"average_rating"`
  RatingCount          int       `bson:"rating_count"`
  CompletionRate       float64   `bson:"completion_rate"`      // 0-1
  AveragePrepMinutes   int       `bson:"average_prep_minutes"`
  PriceBand            int       `bson:"price_band"`           // 1-4 ($-$$$$)

OpenSearch composite ranking spec (function_score):

  weights: price 0.4, quality 0.3, speed 0.2, reliability 0.1
  hybrid retrieval: BM25 over (name, description, cuisine_tags) + KNN over
  restaurant embeddings, combined with Reciprocal Rank Fusion.

  Hard rules:
    - rating_count < 20  → quality_score = 0.5 (neutral)
    - average_rating < 3.0 → EXCLUDE from results
    - completion_rate < 0.80 → multiplicative penalty *0.7
    - price filter is NEVER exposed to user (algorithm-only)

Pseudocode handler:

  func (c *Controller) Search(ctx *fiber.Ctx) error {
      q := ctx.Query("q")
      lat, _ := strconv.ParseFloat(ctx.Query("lat"), 64)
      lng, _ := strconv.ParseFloat(ctx.Query("lng"), 64)
      limit := parseLimit(ctx, 25, 50)
      offset := parseOffset(ctx)
      results, total, err := c.svc.HybridSearch(ctx.Context(),
          q, lat, lng, limit, offset)
      if err != nil { return fiber.NewError(500, err.Error()) }
      return ctx.JSON(fiber.Map{
          "restaurants": results, "total": total,
          "limit": limit, "offset": offset,
      })
  }

If OPENSEARCH_URL env var is unset, the service must fall back to a Mongo
aggregation pipeline that produces equivalent ranking with $function_score-like
$addFields and $sort stages. Customer app must work in either mode.

B.2  PER-ORDER TRANSACTION FEE TRACKING ($0.99/order)
-----------------------------------------------------
Backend has billing scaffolding but does NOT track per-order fees or current

  { _id, restaurant_id, period_start, period_end, order_count,

Hook into order creation (internal/apps/order/service/service.go, after the

  go billingUsageSvc.RecordOrder(ctx, restaurantID, 0.99)

  func (s *Service) RecordOrder(ctx context.Context, rid primitive.ObjectID,
                                fee float64) error {
      now := time.Now().UTC()
      ps := time.Date(now.Year(), now.Month(), 1, 0,0,0,0, time.UTC)
      pe := ps.AddDate(0, 1, 0)
      _, err := s.repo.coll.UpdateOne(ctx,
          bson.M{"restaurant_id": rid, "period_start": ps},
          bson.M{
              "$inc": bson.M{"order_count": 1, "per_order_fee_total": fee},
              "$setOnInsert": bson.M{"period_end": pe, "currency": "usd",
                                     "created_at": now},
              "$set": bson.M{"updated_at": now},
          },
          options.Update().SetUpsert(true),
      )
      return err
  }

================================================================================
NON-NEGOTIABLES
================================================================================

  * BLoC stays. Riverpod is forbidden.
  * Every API call goes through ApiClient → Dio → SAVORAR_API_BASE_URL.
  * Tokens live in flutter_secure_storage only. Never SharedPreferences.
  * No Firebase project, no firebase_* packages in v1.
  * No price-filter UI ever. Composite ranking handles it.
  * Restaurants with rating < 3.0 are excluded server-side. Client trusts the
    server's filtering.
  * The AI chat and AI search endpoints must degrade gracefully if LLM_API_KEY
    is unset — never surface a 500 to the user.
  * All money values come from the server (subtotal, delivery_fee, total).
    The client never recomputes totals.
  * On 401 from any endpoint, the auth interceptor clears the token, emits
    AuthBloc.LoggedOut, and go_router redirects to /welcome.
